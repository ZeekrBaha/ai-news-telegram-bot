# Telegram AI News Aggregator Bot - Design Spec

Date: 2026-05-13  
Author: Baha  
Status: Revised - ready for implementation planning

## 1. Purpose

Build a Python service that runs once per day, collects fresh AI-related news from curated RSS feeds and Telegram channels, ranks the most important items with an LLM, summarizes and adapts them into Russian, and publishes a single daily digest post to a Telegram channel.

The service is autonomous after setup. It keeps a complete audit trail in Supabase so every collection, ranking decision, generated summary, and published digest can be inspected later.

## 2. Goals and Non-Goals

### Goals

- Publish one digest at a configured time, default `09:00 Europe/Moscow`.
- Publish up to 5 items per digest, ranked by relevance and importance.
- Allow a configurable minimum item count, default `3`, so good digest days are not skipped just because fewer than 5 unique items exist.
- Produce Russian-language output with a title, 3-5 bullets, a "why this matters" note, source links, and hashtags.
- Deduplicate against all previously seen items before publishing.
- Read Telegram channels through Telethon using a user session.
- Publish through the Telegram Bot API.
- Store raw items, run state, ranking output, processed output, and digest metadata in Supabase.
- Support dry runs that format the digest without publishing.

### Non-Goals

- Human moderation in the MVP.
- Reddit, X, or direct Hacker News API collectors. HN may be consumed through RSS.
- Multiple output channels.
- Web admin UI.
- External monitoring such as Sentry.
- Engagement-based ranking.

## 4. Architecture

One Dockerized Python service runs continuously. APScheduler fires a daily job. Supabase stores all durable state. The container is stateless except for a mounted Telethon session directory.

The async pipeline is run through one clear boundary: the scheduler job calls `asyncio.run(run_daily())`. Inside the pipeline, RSS and Telegram collectors are async. Supabase access can remain sync if wrapped cleanly in repository functions.

```text
scheduler
  -> pipeline orchestrator
      -> collectors (rss, telegram)
      -> canonicalizer
      -> pre-insert deduplicator
      -> raw item writer
      -> ranker
      -> summarizer / translator
      -> digest formatter
      -> pending digest writer
      -> Telegram publisher
      -> digest finalizer
      -> run finalizer
```

## 6. Core Data Types

```python
@dataclass(frozen=True)
class CollectedItem:
    source_type: str          # "rss" | "telegram"
    source_name: str
    source_item_id: str       # feed entry id, Telegram message id, or stable synthetic id
    url: str | None
    canonical_url: str | None
    url_hash: str             # sha256(canonical_url or source identity)
    title_hash: str           # sha256(normalized title)
    title: str
    content: str              # plain text, truncated for LLM input
    published_at: datetime
    raw: dict
```

`RankedItem`, `SummarizedItem`, and `TranslatedItem` extend this base shape with stage-specific fields. JSON generated by the LLM must be validated into typed Pydantic models before being used by later stages.

## 7. Deduplication Design

Deduplication must happen before raw insert, or the run must explicitly distinguish rows that existed before the current run from rows inserted during it. The MVP uses pre-insert dedupe:

1. Collect candidate items.
2. Normalize each item into `CollectedItem`.
3. Compute `url_hash` from canonical URL when available.
4. For Telegram/private/no-URL items, compute `url_hash` from `source_type`, `source_name`, and `source_item_id`.
5. Compute `title_hash` from lowercased, punctuation-collapsed title.
6. Query Supabase for existing `url_hash` and `title_hash`.
7. Keep only items that do not match prior rows.
8. Insert only kept items into `raw_items`.

This avoids the original bug where inserting rows first makes the dedupe query see the current run's new rows as if they were old rows.

## 8. Supabase Schema

All tables use `id uuid primary key default gen_random_uuid()` and `created_at timestamptz default now()`.

### `runs`

```sql
started_at timestamptz default now()
finished_at timestamptz
status text not null -- running | success | skipped | failed
error text
items_collected int default 0
items_after_dedup int default 0
items_published int default 0
```

### `raw_items`

```sql
run_id uuid references runs(id)
source_type text not null
source_name text not null
source_item_id text not null
url text
canonical_url text
url_hash text not null unique
title_hash text not null
title text not null
content text
published_at timestamptz
raw jsonb
collected_at timestamptz default now()
```

Indexes:

```sql
create unique index raw_items_url_hash_key on raw_items(url_hash);
create index raw_items_title_hash_idx on raw_items(title_hash);
create index raw_items_collected_at_idx on raw_items(collected_at);
create index raw_items_source_idx on raw_items(source_type, source_name, source_item_id);
```

### `ranked_items`

```sql
raw_item_id uuid references raw_items(id)
run_id uuid references runs(id)
rank int not null
score float
reasoning text
unique (run_id, raw_item_id)
unique (run_id, rank)
```

### `processed_items`

```sql
ranked_item_id uuid references ranked_items(id) unique
summary_en text
title_ru text not null
bullets_ru text[] not null
why_it_matters_ru text
hashtags text[]
validation_notes text
```

### `digests`

```sql
run_id uuid references runs(id) unique
status text not null -- pending | published | failed
content_hash text not null
telegram_message_id bigint
channel_id text not null
posted_at timestamptz
item_ids uuid[] not null
error text
```

The digest row is created with `status='pending'` before publishing. After Telegram returns a message id, update the row to `published`. If Telegram returns an error, update it to `failed`. If the network times out after a possible send, do not automatically retry; inspect the channel and row manually.

## 9. Pipeline Logic

1. Create a `runs` row with `status='running'`.
2. Collect RSS and Telegram items.
3. Normalize and canonicalize each item.
4. Drop items older than `MAX_AGE_HOURS`, default 36.
5. Pre-dedupe against `raw_items.url_hash` and `raw_items.title_hash`.
6. Insert only new raw items, tied to the current `run_id`.
7. If fewer than `MIN_DIGEST_ITEMS` remain, finalize the run as `skipped` and do not publish.
8. Rank candidates in one LLM call and keep up to `DIGEST_TOP_N`, default 5.
9. Summarize and translate each selected item. Validate every model response.
10. Build the digest string.
11. If the digest is too long, shorten `why_it_matters_ru`, then bullets, then remove lowest-ranked items until it fits.
12. Create a pending digest row with content hash and item ids.
13. Publish to Telegram once.
14. Update the digest row with `telegram_message_id` and `status='published'`.
15. Finalize the run with counts and status.

## 10. Failure Handling

| Failure | Behavior |
| --- | --- |
| One RSS feed times out | Warn and continue with other sources. |
| Telethon auth fails | Log error and continue RSS-only. |
| OpenAI ranking fails after retries | Mark run failed; publish nothing. |
| One summarization/translation fails | Retry. If still invalid, use degraded fallback only if at least the source title and URL are available. |
| Too few items | Mark run skipped; publish nothing. |
| Telegram publish fails definitively | Mark digest and run failed; do not auto-retry. |
| Telegram publish times out ambiguously | Mark digest failed with ambiguity note; require manual channel check. |
| Unexpected exception | Catch at top level, store traceback in `runs.error`, keep scheduler alive. |

## 11. Telegram Formatting

Use Telegram HTML parse mode unless there is a strong reason to use MarkdownV2. HTML is easier to escape reliably for generated Russian text and source links.

Allowed generated tags should be minimal: `<b>`, `<i>`, and `<a href="...">`. All text is escaped with `html.escape`. URLs are validated before insertion.

If MarkdownV2 is chosen later, it must have dedicated tests for every reserved character and link target escaping.

## 12. Observability

- Log to stdout at `INFO`.
- Every run writes a row to `runs`.
- Every raw item and processed item is stored for audit.
- Ranking reasoning is stored for prompt tuning.
- A dry run prints the final digest and writes no published digest row unless explicitly configured.
- External monitoring is out of MVP scope, but the design leaves room for Sentry later.

## 14. Security

- Never commit `.env`, `sessions/*.session`, Supabase service-role keys, Telegram tokens, or OpenAI keys.
- `.gitignore` must include `.env`, `.env.*`, `sessions/`, `telethon_session/`, and local database dumps.
- The Supabase service-role key is server-only.
- The Telethon session file is equivalent to account access and must be stored with restrictive permissions.
- Document a manual secret rotation path for Telegram bot token, OpenAI key, Supabase service key, and Telegram user session.

## 16. Testing Strategy

- Unit tests for canonicalization, hash generation, dedupe decisions, RSS parsing, Telegram item conversion, formatter escaping, and prompt builders.
- AI tests mock OpenAI and validate schema parsing, bad JSON handling, retry behavior, and fallback behavior.
- Repository tests verify pre-insert dedupe and unique `url_hash` behavior.
- Integration test runs one complete pipeline with mocked OpenAI and Telegram against a Supabase test schema.
- Manual smoke test runs `python -m src.main --once --dry-run` for the first few days before enabling production publishing.
